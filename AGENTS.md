# Воркхабы для театра — обязательные инструкции для Codex

## Прочитать до начала работы

Это действующий рабочий проект камерного театра-лаборатории Т.А.М. Перед любой работой:

1. Прочитай этот файл полностью.
2. Кратко изучи только те файлы, которые относятся к новой задаче. Не перечитывай весь проект ради небольшой правки.
3. Не начинай менять код, данные, базу или файлы, пока не выполнишь порядок согласования из раздела «Как начинать каждую задачу».

## Что уже есть

Воркхаб — единое приложение театра с доступом по ролям. Уже реализованы или начаты: главный экран разделов, участники и настройки, управление разделами, копилка материалов, календарь репертуара, расписание классов и репетиций, контент-план, загрузка файлов, аватары, роли и права, корзина, PWA и Supabase.

Доступы не должны быть «для вида»: недоступный раздел показывается на главном экране серым с надписью «Нет доступа» и не должен открываться. Новые разделы должны появляться в настройках ролей автоматически.

Роли: разработчик, руководитель, педагог, админ, участник. Точные права всегда проверяй по текущим данным и миграциям, не угадывай их.

Календарные данные, расписание, файлы, аватары и контент-план — пользовательские данные. К ним относиться особенно бережно.

## Неприкосновенность данных

- Ничего пользовательского не удалять без явного подтверждения пользователя: записи, файлы, вложения, участников, разделы, роли и данные Supabase.
- Удаление в интерфейсе всегда должно требовать понятного подтверждения.
- Корзина очищается автоматически через 30 дней. Не меняй это правило без прямой просьбы.
- Не выполнять `git reset --hard`, `git checkout --`, массовые удаления и не перезаписывать чужие изменения.
- Не удалять и не менять исходный демонстрационный раздел, который пользователь просил пока не трогать, без отдельной явной просьбы.
- Любая миграция должна сохранять существующие записи и быть обратимо-безопасной настолько, насколько возможно.
- Не раскрывать ключи, пароли, токены, содержимое `.env.local` и другие секреты в ответах или коммитах.

## Визуал и мобильная версия

- Не менять дизайн или функционал вне конкретной задачи.
- Шапка должна оставаться закреплённой, чёрный фон должен доходить до верхнего края экрана и учитывать вырез камеры iPhone.
- Иконки придерживаются единого чёрного стиля, если пользователь явно не попросит другое.
- Уведомления не должны сдвигать интерфейс: использовать всплывающий слой.
- Проверять и компьютерную, и мобильную версию. На телефоне не допускать обрезки важных текстов, наездов на счётчики или недоступной горизонтальной прокрутки.
- Все переходы и календарные переключения должны выглядеть плавно, без резких скачков.

## Как начинать каждую задачу — обязательный порядок

Пользователь использует ChatGPT Plus и бережёт недельный лимит Codex.

До любой реализации сначала:

1. Коротко проанализируй задачу: что именно будет затронуто и есть ли риск для данных, ролей или синхронизации.
2. Напиши рекомендацию по настройкам в одном коротком сообщении: модель, усилие, скорость.
3. Остановись и дождись, пока пользователь ответит, что включил настройки (например: «включил», «готово», «давай»).
4. Только после этого начинай менять проект.

Шаблон первого сообщения:

> Для этой задачи рекомендую: **[модель] → [усилие] → Стандартная скорость**. Затронем: [кратко]. Включите, пожалуйста, и напишите «готово» — после этого начну работу.

Исключение: если пользователь прямо и однозначно попросил не ждать подтверждения в этой конкретной задаче, можно работать сразу, но всё равно сообщить выбранные настройки.

### Выбор настроек

| Вид задачи | Примеры | Рекомендация |
| --- | --- | --- |
| Мелкая понятная правка | Текст, подпись, отступ, цвет, иконка, один элемент на мобильном экране | Terra → лёгкое → стандартная |
| Обычная доработка экрана | Вкладка, карточка, форма, локальная адаптация телефона | Terra → среднее → стандартная |
| Календарь, файлы, роли, синхронизация, новая функция с данными | Раздел календаря, контент-план, права доступа, загрузка файлов | Sol → среднее → стандартная |
| Непонятная ошибка или риск для данных | Supabase, RLS, JWT, пропадают записи, неверный доступ | Sol → высокое → стандартная |
| Срочная маленькая правка перед показом | Одна конкретная ошибка, когда важна каждая минута | Terra или Sol по сложности → лёгкое/среднее → Fast только по явной необходимости |

Если Terra отсутствует в выборе модели, использовать Sol с лёгким или средним усилием по таблице. Не включать Fast по умолчанию: он экономит время ожидания, но расходует лимит быстрее.

## Как работать после подтверждения

1. Проверь состояние Git и изучи только относящиеся к задаче файлы.
2. Сделай минимально необходимое изменение. Не проводи «заодно» рефакторинг, редизайн или чистку.
3. Если задача требует изменения Supabase, подготовь отдельную миграцию и примени её только в согласованном объёме.
4. Проверь сборку командой `pnpm run build`.
5. Проверь результат в приложении в объёме задачи, включая мобильную версию, если затронут интерфейс.
6. Опубликуй подтверждённые изменения в `main` и проверь рабочую ссылку GitHub Pages.
7. В финале коротко скажи: что сделано, что проверено, что не менялось и есть ли особенность для пользователя.

## Общение с пользователем

- Писать по-русски, простыми словами, без лишней технической терминологии и длинных лекций.
- Пользователь быстро печатает и может писать с ошибками или без русской раскладки. Понимай по контексту; переспрашивай только если неоднозначность реально меняет результат или может затронуть данные.
- Не отвечать общими обещаниями. Сообщать конкретный результат и проверку.
- Не требовать от пользователя искать папки, файлы, консоли или настройки, если это можно проверить самостоятельно.
- Пользователь хочет сам тестировать всё по одной рабочей ссылке. После изменений публиковать туда, если задача не ограничена только обсуждением.
- При пропадании связи продолжать с сохранённым контекстом и не терять принятые решения.

## Лимиты и организация работы

- Не тратить Sol и высокое усилие на простые визуальные изменения.
- Не включать Fast без необходимости.
- Для следующего крупного самостоятельного этапа лучше начать новый диалог, но в той же папке проекта: этот файл и Git сохраняют правила и состояние.
- В новом диалоге сначала прочитать `AGENTS.md`, затем оценить задачу и дождаться подтверждения настроек.
